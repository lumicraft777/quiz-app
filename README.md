# 蓄電池メーカー比較クイズ｜営業判断力トレーニングアプリ（Supabase連携版）

Googleスプレッドシートのリサーチデータから自動生成した4択クイズを、Supabase（DB）経由で配信するWebアプリです。ユーザーの成績（自己ベスト・コンボ・要再挑戦リスト）もSupabaseに保存されるため、**スマホ・PCなどどの端末からでも、プレイヤーID＋認証キーでログインすれば同じ記録を引き継げます**。

見た目は「近未来の仮想訓練空間で任務を攻略していくRPG」の世界観で統一されています（クイズ＝任務、カテゴリ＝エリア、連続正解＝コンボ、ランキング＝攻略者ボード、等）。詳しくは[世界観・ゲーミフィケーション機能](#世界観ゲーミフィケーション機能)を参照してください。

- 公開URL：https://lumicraft777.github.io/quiz-app/
- アプリ本体：[index.html](index.html) / [style.css](style.css) / [script.js](script.js)
- 問題生成ロジック（Node専用）：[question-generator.js](question-generator.js)
- 同期スクリプト：[scripts/sync-questions.js](scripts/sync-questions.js)
- DBスキーマ：[supabase/schema.sql](supabase/schema.sql)

---

## 全体の仕組み

```
Googleスプレッドシート（元データ）
   │  scripts/sync-questions.js を実行（手動・またはデータ更新のたびに）
   ▼
Supabase: questions テーブル（生成済みの4択問題）
   │  ブラウザがこのテーブルを直接読む（publishable keyのみ使用）
   ▼
ブラウザ（index.html / script.js） ← GitHub Pagesで配信
   │  ユーザー名＋PINでログイン（rpc_login）
   │  回答するたびに記録（rpc_record_answer）
   ▼
Supabase: users / answer_history テーブル（ユーザーごとの成績）
```

以前（スプレッドシート直読み版）と違い、**ブラウザは起動のたびにスプレッドシートを読みに行きません**。スプレッドシートを更新したら、`scripts/sync-questions.js` を実行してSupabase側の`questions`テーブルを作り直す必要があります。

## 使い方（実行手順）

```bash
# battery-quiz-app フォルダ内で実行（Pythonが入っていればOK）
python -m http.server 8000
```

その後、ブラウザで `http://localhost:8000` を開きます（本番は上記の公開URLでそのままアクセスできます）。

1. 起動すると「MISSION DIVE」画面でプレイヤーID＋認証キー（4桁以上の数字）の入力を求められます。初めてのIDならそのキーで新規登録され、既存のIDなら同じキーでダイブ（ログイン）します（キーが違うと拒否されます＝他人が同じIDを使っても成績を見られません）
2. データはバックグラウンドで自動同期され、「任務を開始」ボタンの下に控えめな同期状況（任務数の内訳・要再挑戦リスト数）が表示されます
3. 任務モード・エリア・レベル・任務規模を選んで「任務を開始」
4. スタート画面の「攻略者ボードを見る」で全ユーザー横断の成績を確認できます

## スプレッドシートを更新したときにやること

1. スプレッドシートの内容を編集する（列構成は変えないこと。詳しくは後述）
2. `battery-quiz-app` フォルダで以下を実行する

```bash
node scripts/sync-questions.js
```

これでSupabaseの`questions`テーブルが最新のスプレッドシート内容から作り直されます（全件削除→再生成なので、問題IDが振り直されます。既存の不正解/正解済みリストのうち、削除された問題IDに紐づく回答履歴は自動的に消えます＝外部キーのON DELETE CASCADEによる）。アプリ側の再デプロイは不要で、次にブラウザを開いた/再読み込みした瞬間から新しい問題が使われます。

### 初回セットアップ（Supabaseプロジェクトを作り直す場合）

1. [supabase.com](https://supabase.com) で無料プロジェクトを作成（リージョンは Northeast Asia (Tokyo) 推奨）
2. SQL Editorで [supabase/schema.sql](supabase/schema.sql) の中身を実行
3. `battery-quiz-app/.env.example` を `.env` にコピーし、Project Settings → API で確認できる値を入れる
   ```
   SUPABASE_URL=https://xxxxx.supabase.co
   SUPABASE_SECRET_KEY=sb_secret_...
   ```
   **`.env` は`.gitignore`済みです。絶対にコミット・共有しないでください。**
4. `node scripts/sync-questions.js` を実行して問題を流し込む
5. [script.js](script.js) 冒頭の `SUPABASE_URL` / `SUPABASE_PUBLISHABLE_KEY` を新プロジェクトの値に書き換える（publishable keyはブラウザに公開しても安全な設計なので、コード中に直接書いてOK）

## セキュリティモデル（重要）

- **publishable key**（`script.js`に直書き）：ブラウザに公開される前提のキー。RLS（Row Level Security）で保護されており、`questions`テーブルの読み取りと、決められたRPC関数の呼び出ししかできません
- **secret key**（`.env`のみ）：RLSを無視できる管理者権限のキー。`scripts/sync-questions.js`以外では絶対に使わないこと。チャットや公開リポジトリに貼らないこと
- `users`/`answer_history`テーブルはRLSで直接アクセス禁止にしてあり、すべて`rpc_login`・`rpc_record_answer`などのSECURITY DEFINER関数経由でのみ操作できます（PINハッシュや他人のuser_idがクライアントから直接読めない設計）
- 認証はユーザー名＋4桁PINの軽量な仕組みです。メールアドレス等を使う本格的な認証ではないため、社内の小規模な営業チームなど「信頼できるメンバーだけが使う」前提の運用を想定しています

## ユーザーごとの成績・不正解/正解済みリスト

- Supabaseの`users`テーブル（ユーザー名＋PINハッシュ＋自己ベスト等の集計値）と`answer_history`テーブル（1回ごとの回答ログ）で管理
- 「現在の不正解/正解済みリスト」は、`answer_history`の**各問題ごとの最新の回答結果**から毎回計算します（正解すればそのIDは不正解リストから消え、不正解リストに移る＝どちらか一方にしか入らない）
- カテゴリ選択に「不正解問題（◯問）」「正解問題（◯問）」が現れるのは、該当リストに1問以上あるときだけです
- 結果画面の「あなたの記録」は常に**現在ログイン中のユーザーのみ**の成績です
- ランキング画面は全ユーザー横断ですが、ユーザー名と成績（正答率・正答数・連続正解・累計）以外の情報（PIN等）は一切表示されません

## スプレッドシートのデータ形式

Claude in Chromeのリサーチ結果（タブ区切りの表）を、**1行目が見出し行になるように**シートへそのまま貼り付けてください。同期スクリプトは以下を自動処理します。

- 見出し行は「メーカー名」列を探して自動検出（先頭に多少のメモ行があってもOK）
- メーカー名・製品名が空の行、表の下の「ランキング」などのメモ行、貼り付け時に混ざった2つ目の見出し行は自動スキップ
- セル内のカンマ・改行・引用符にも対応
- 列の**並び順が変わっても見出し名で照合**するので問題なし（見出し名自体は変えないでください）

### 問題化されないデータの扱い（重要）
- 値が「不明」「未確認」「公式未確認」「判断保留」（を含む文字列）・空欄・URLのみの項目は**その項目の問題を生成しません**
- ダミー選択肢（他製品の異なる値）が3つ揃わない項目も生成されません

## 問題の自動生成の仕組み（question-generator.js）

### レベル（初級/中級/上級）の位置づけ
| レベル | 位置づけ | 内容 |
|---|---|---|
| 初級 | 訪問営業のアポインターが最低限覚えるべき基礎知識 | `generateBasicConceptQuestions()`が生成する、スプレッドシートに依存しない静的な33問。太陽光/蓄電池の役割の違い、停電時の基本、全負荷・特定負荷の超基本、V2Hの超基本、お客様からよく聞かれる質問への回答など。カテゴリは「基礎知識」に統一 |
| 中級 | 商談前に理解しておきたいメーカー・製品比較の基本 | シートの各製品データから自動生成する値当て・逆引き・あり/なし問題（保証年数・容量・電池材料・負荷タイプ・V2H対応・設置条件など） |
| 上級 | 営業現場で提案判断ができるレベル | お客様状況カードをもとにした実践提案問題、営業トーク消去法問題、複数メーカー比較（最大/最小値当て）問題 |

初級は細かい型番・数値の暗記ではなく「意味・違い・用途」を問う内容にとどめ、型番レベルの知識比較は中級以降に位置づけている。

### 知識問題（中級）
| 種類 | 例 | カテゴリ |
|---|---|---|
| 値当て | 「◯◯の製品保証年数は？」「蓄電容量は？」「電池材料は？」「負荷タイプは？」「停電時出力は？」「太陽光連携は？」「設置条件は？」「設計寿命は？」「蓄電池タイプは？」 | 保証/容量/電池材料/停電対策/太陽光連携/メーカー比較 |
| 逆引き | メリット・デメリット・特徴・向いている家庭・営業訴求ポイントの文章から製品を当てる（選択肢は必ず「メーカー＋製品名」まで表示） | メリット/デメリット・営業トーク・メーカー比較 |
| あり/なし | 「V2H対応しているのは？」「自然災害補償が付帯していないのは？」「AI/HEMS対応は？」 | V2H/保証/メーカー比較 |
| 比較（上級） | 「蓄電容量が最も大きいのは？」「保証が最も長いのは？」 | 保証/メーカー比較 |

解説には「主な特徴」「メリット」「営業時の訴求ポイント」のうちシートに値がある要素を必ず含めます。選択肢ごとにも「正解/不正解」「理由（誤答は実際にどの製品の値かを明示）」を`choiceExplanations`として保持し、解説画面で4択すべてを比較表示します。

### 実践提案問題（上級）
シートの「向いている家庭」「営業時の訴求ポイント」「デメリット」列を材料に自動生成します。

1. 各製品の文章をキーワードスコアリングし、重視カテゴリ（停電対策/電気代削減/初期費用/EV・V2H/保証・安心/設置スペース）を推定
2. **製品選択型**：お客様状況を提示し、最適な製品を選ぶ。誤答は実データ上の客観的な失格条件（`PRACTICE_DISQUALIFIER_RULES`）を優先的に使う
3. **営業トーク消去法**：「明らかに不適切な提案はどれ？」形式。失格条件を実データで確認できる場合のみ出題する
4. 解説は「なぜ正解か／判断材料になった条件／他の選択肢が弱い理由／営業時の伝え方／注意点」の5要素で構成

## 経験値（EXP）・レベル機能

正解・不正解・解説閲覧・連続正解（コンボ）・10問完走のたびにEXPを獲得し、レベルアップします。5レベルごとに称号が変わり、Lv.100が上限（MAX LEVEL）です。

- 必要EXP：`50 × 現在レベル`（複数レベル分のEXPが一度に入った場合は複数レベルアップします）
- EXP獲得ルール：正解+10（復習モードは+15）／不正解+3（不正解でも学習したとみなす）／解説閲覧+5／10問セッション完走+20（セッション内1回のみ）／コンボボーナス（3・5・10・15・20連続正解でそれぞれ+5・+10・+25・+40・+60、いずれもセッション内1回のみ）
- クイズ中はクライアント側（`state.sessionExp`等）でEXPを積算するだけにしておき、**結果画面表示時に合計獲得EXPをまとめてSupabaseへ反映**します（`rpc_apply_exp`）。通信回数を抑えるための設計で、途中退出した場合もそれまでに回答した分のEXPだけが正しく反映されます
- レベルアップ時は結果画面で演出（LEVEL UPアニメーション）が表示され、5レベル境界をまたいで称号が変わった場合は「🏆 TITLE UPGRADE！🏆」の特別演出になります
- EXPゲージは開始画面・クイズ画面・結果画面・ランキング画面に表示されます（称号・現在EXP/必要EXP・残りEXP・達成率）

## 世界観・ゲーミフィケーション機能

アプリ全体を「近未来の仮想訓練空間で任務を攻略していくRPG」の世界観で統一しています。用語の対応関係は以下の通りです。

| 通常のクイズアプリの用語 | このアプリでの表現 |
|---|---|
| クイズ開始 / クイズモード | 任務を開始 / 任務モード |
| 知識問題モード / 実践提案モード / ミックスモード | 基礎任務 / 判断任務 / 混成任務 |
| カテゴリ | エリア |
| 連続正解 | コンボ |
| 不正解問題（復習リスト） | 要再挑戦リスト |
| 解説 | 解析 |
| 用語集 | ワールド辞典 |
| ランキング | 攻略者ボード |
| 結果画面 | 任務結果 |

初級・中級・上級、正答率、不正解、正解問題（正解済みリスト）はそのままの表現を使っています。

### ログイン画面（MISSION DIVE）
ログイン画面は「仮想訓練空間へダイブする入口」として作り込んでいます。ダークなグリッド背景・スキャンライン演出・発光する認証パネルで構成し、「プレイヤーID」「認証キー」を入力して「ダイブ開始」ボタンを押すと、「仮想訓練空間へ接続中...」→「接続完了」の演出を経てスタート画面（任務端末）へ遷移します。`#dive-bg`・`#dive-connecting-overlay`は`#screen-user`の外（bodyの直下）に置いており、画面切り替えの`transform`アニメーションに`position:fixed`の要素が閉じ込められないようにしています。

### 任務結果画面
正答率・正解数に加えて、**任務達成率**（予定問題数のうち実際に回答できた割合。途中中断すると100%未満になる）・**制圧率**（正答率のRPG的な言い換え）・**被弾数**（不正解数）・**最大連撃数**（セッション内の最大コンボ）を表示します。

### 称号（バッジ）
`badges`/`user_badges`テーブルで管理する12種類の称号があります（はじめの一歩・集中モード・復習の鬼・上級アドバイザー・オールラウンダー・継続の達人・完全制圧者・習慣化の入口・訓練兵・攻略者・連撃マスター・再起の達人）。判定はすべて`rpc_check_badges`がSupabase上の永続データ（`users`/`answer_history`/`questions`）だけを根拠にサーバー側で行い、クライアントからの自己申告は一切信用しません。結果画面でEXP反映後に自動判定され、新規解放があれば専用の演出（`#badge-unlock-overlay`）で1つずつ表示します。スタート画面には称号一覧（解放済み/未解放）が常に表示されます。

### デイリーミッション
`daily_missions`/`user_daily_mission_claims`テーブルで管理する、当日中に達成できる4つのミッション（基礎任務5問・判断任務3問・要再挑戦リスト3問・合計10問）があります。進捗は`answer_history`の当日分をその場で集計して求めるため、進捗用の状態を別途持つ必要がありません（`reward_claimed`だけを記録し、達成条件は毎回サーバー側で再検証します）。達成後は「報酬を受け取る」ボタンからEXPを受け取れます。

### 連続ログイン（学習ストリーク）
`users.streak_days`/`last_active_date`で管理します。`rpc_login`内で、前回ログイン日が「昨日」ならストリーク+1、「今日」なら変化なし、それより前（または初回）なら1にリセットします。2日以上のストリークがあるとスタート画面にバナー表示されます。

### 攻略者ボード（複数ランキング）
総合（既存の正答率順ランキングは変更なし）・今週・連撃（最大コンボ）・制圧率・任務数（累計解答数）・復習（復習モードでの正解数）の6種類をタブで切り替えて閲覧できます。総合以外はそれぞれ専用のRPC（`rpc_get_weekly_ranking`等）から取得します。

## 用語集（辞書）機能

画面左上の📖ボタンから、検索欄付きの用語集パネルを開けます。また、問題文・選択肢・解説文の中に用語集に登録された単語が含まれていると自動的に下線付きでハイライトされ、タップするとその用語で辞書パネルが開きます。

- データはSupabaseの`glossary`テーブル（`term`/`definition`/`category`）。RLSで読み取り専用に公開
- 元データは[scripts/glossary-data.js](scripts/glossary-data.js)（初期27語、営業目線の定義文）
- 更新・追加方法：`glossary-data.js`を編集して `node scripts/sync-glossary.js` を実行（questionsと同様、全件削除→再投入）
- 将来スプレッドシートに「用語集」タブを追加してそこから同期したくなった場合は、`sync-glossary.js`を`question-generator.js`と同じパターンで書き換えれば対応できます

## 将来の拡張

- **新ジャンルのシート追加**（補助金・エコキュート等）：`question-generator.js`の`SHEET_CONFIG`にシートを追加し、その`type`用の生成関数を書く。生成した問題は同じ`questions`テーブルに`mode`/`category`を変えて入れられる
- **用語集のスプレッドシート化**：現在は`glossary-data.js`に直接記述。件数が増えてきたらスプレッドシート管理に移行できる
- **セキュリティ強化**：現在はユーザー名＋PINの軽量認証。将来的にSupabase Auth（メール認証等）へ移行することも可能

## 注意事項

- `.env`・secret keyは絶対にコミット・チャット共有しないこと。誤って共有した場合はSupabaseダッシュボードでキーをローテーションすること
- 問題の正確性はスプレッドシートのデータの正確性に依存します。同期後は`node scripts/sync-questions.js`の出力（生成件数）を確認してください
